# 数据表结构与关系文档

## 1. 文档说明

本文档基于以下内容整理：

- `CeuimDataScreenSpringboot/src/main/resources/init.sql`
- `CeuimDataScreenSpringboot/src/main/resources/danger_work_init.sql`
- 后端 `entity` 实体类

目标：

- 说明当前数据库核心表结构
- 明确主键、外键、逻辑关联关系
- 标记空间字段与业务模块映射
- 为后续数据库迭代、接口开发、页面联调提供基线

数据库类型：

- PostgreSQL

空间扩展：

- PostGIS

数据库名称：

- `cityplan`

## 2. 数据库整体定位

当前数据库大体可分为四层：

### 2.1 用户与权限层

- `sys_user`

### 2.2 业务主数据层

- `biz_project`
- `biz_land_reserve`
- `biz_property`
- `biz_dangerous_work`
- `biz_project_file`

### 2.3 配置驱动层

- `sys_dict`
- `sys_dict_item`
- `sys_form_template`

### 2.4 地图与扩展层

- `map_drawing`
- `sys_log`
- `biz_danger_point`

说明：

- `biz_danger_point` 出现在 `init.sql` 中，偏向较早期设计
- 实际危大作业当前核心模块使用的是 `biz_dangerous_work`
- 这说明数据库中存在一定的历史设计痕迹

## 4. 核心表结构

### 4.1 `sys_user`

用途：

- 系统用户主表
- 存储登录账户、密码、角色、启用状态

主键：

- `id`

关键字段：

- `username`
- `password`
- `email`
- `phone`
- `real_name`
- `avatar`
- `role`
- `enabled`
- `created_at`
- `updated_at`
- `deleted`

约束与说明：

- `username` 唯一
- `role` 当前取值为：
  - `USER`
  - `ADMIN`
  - `SUPER_ADMIN`
- 使用逻辑删除字段 `deleted`

关联关系：

- `biz_project.created_by -> sys_user.id`
- `map_drawing.created_by -> sys_user.id`
- `sys_log.user_id -> sys_user.id`
- `biz_project_file.uploaded_by` 逻辑上关联 `sys_user.id`

### 4.2 `sys_dict`

用途：

- 存储字典类型
- 为项目分类、业务分类、筛选项等提供配置入口

主键：

- `id`

关键字段：

- `dict_code`
- `dict_name`
- `description`
- `created_at`
- `updated_at`
- `deleted`

约束与说明：

- `dict_code` 唯一
- 使用逻辑删除字段 `deleted`

关联关系：

- `sys_dict_item.dict_id -> sys_dict.id`

### 4.3 `sys_dict_item`

用途：

- 存储字典项

主键：

- `id`

关键字段：

- `dict_id`
- `item_code`
- `item_name`
- `parent_id`
- `sort`
- `color`
- `status`
- `created_at`
- `deleted`

约束与说明：

- `dict_id` 为物理外键
- `parent_id` 当前用于层级字典扩展
- `color` 可用于地图颜色展示
- 使用逻辑删除字段 `deleted`

典型数据示例：

- `project_category`
  - `LAND`
  - `ENGINEERING`
- `land_sub_type`
  - `RESIDENTIAL`
  - `COMMERCIAL`
  - `INDUSTRIAL`
- `engineering_sub_type`
  - `DAIJIAN`
  - `GONGJIAN`
  - `SHIZHENG`
  - `FANGDICHAN`
  - `PIANQUGUIHUA`

### 4.4 `sys_form_template`

用途：

- 动态表单模板定义表
- 按 `category + sub_type` 对应项目扩展字段

主键：

- `id`

关键字段：

- `template_name`
- `category`
- `sub_type`
- `fields`
- `created_at`
- `updated_at`
- `deleted`

约束与说明：

- `(category, sub_type)` 唯一
- `fields` 为 `JSONB`
- 使用逻辑删除字段 `deleted`

逻辑关联关系：

- `biz_project.category + biz_project.sub_type`
  对应
- `sys_form_template.category + sys_form_template.sub_type`

### 4.5 `biz_project`

用途：

- 项目主业务表
- 是当前系统最核心的业务表

主键：

- `id`

关键字段：

- `project_code`
- `project_name`
- `project_type`
- `category`
- `sub_type`
- `status`
- `start_date`
- `end_date`
- `description`
- `address`
- `district`
- `area`
- `geometry`
- `extra_data`
- `created_by`
- `created_at`
- `updated_at`
- `deleted`

扩展字段：

- `total_investment`
- `secondary_group`
- `construction_unit`
- `development_unit`
- `fund_source`
- `procedures_status`
- `completion_procedures`
- `existing_problems`
- `current_investment`

字段说明：

- `category`
  - 当前主要用于区分项目大类
  - 实际代码里出现过：
    - `LAND`
    - `ENGINEERING`
    - `DISTRICT`
- `sub_type`
  - 子类编码
- `project_type`
  - 当前前端部分场景下保存的是子类名称或展示名称
  - 这意味着 `sub_type` 与 `project_type` 存在“编码 / 展示值并存”的情况
- `extra_data`
  - `JSONB`
  - 用于存储动态表单扩展字段
- `geometry`
  - PostGIS 空间字段
  - SRID 为 `4326`

约束与说明：

- `created_by` 为物理外键，关联 `sys_user.id`
- 使用逻辑删除字段 `deleted`
- 当前未在 SQL 中显式声明 `project_code` 唯一，但业务层有唯一性校验

索引：

- `idx_project_geometry`，GIST 空间索引

关联关系：

- `biz_project.created_by -> sys_user.id`
- `biz_project_file.project_id -> biz_project.id`（逻辑关联）
- `biz_danger_point.project_id -> biz_project.id`
- `biz_dangerous_work.district_project_id -> biz_project.id`
- `map_drawing.business_id` 在 `category=PROJECT` 时逻辑上关联 `biz_project.id`

## 5. 关系结构说明

## 5.1 物理外键关系

当前数据库中明确存在的外键关系包括：

- `biz_project.created_by -> sys_user.id`
- `map_drawing.created_by -> sys_user.id`
- `sys_log.user_id -> sys_user.id`
- `sys_dict_item.dict_id -> sys_dict.id`
- `biz_danger_point.project_id -> biz_project.id`
- `biz_dangerous_work.district_project_id -> biz_project.id`

## 5.2 代码层逻辑关联

以下关系在业务上成立，但未全部做成数据库物理外键：

- `biz_project_file.project_id -> biz_project.id`
- `biz_project_file.uploaded_by -> sys_user.id`
- `biz_project.category + biz_project.sub_type`
  对应
  `sys_form_template.category + sys_form_template.sub_type`
- `biz_project.sub_type`
  逻辑上来自
  `sys_dict_item.item_code`
- `map_drawing.business_id`
  根据 `category` 指向不同业务表

## 5.3 关系图概念说明

可用文字关系概括为：

- 一个用户可以创建多个项目
- 一个用户可以创建多个地图标绘
- 一个用户可以上传多个项目附件
- 一个字典类型包含多个字典项
- 一个项目可以有多个附件
- 一个项目可以关联多个危大作业
- 一个项目、土地或房产都可以挂多个地图标绘
- 一个项目类别与子类可匹配一个动态表单模板

## 6. 空间数据设计

当前系统中存在两种空间数据存储方式：

### 6.1 PostGIS 几何字段

适用于业务主表：

- `biz_project.geometry`
- `biz_land_reserve.geometry`
- `biz_property.geometry`
- `biz_danger_point.location`

特点：

- 可建立 GIST 空间索引
- 适合查询、范围分析、空间运算

### 6.2 GeoJSON 文本字段

适用于地图扩展表：

- `map_drawing.geometry`

特点：

- 存储灵活
- 前端直接读写方便
- 但数据库层空间分析能力弱于 PostGIS 几何字段

### 6.3 经纬度字段

危大作业当前使用：

- `biz_dangerous_work.lng`
- `biz_dangerous_work.lat`

特点：

- 前端直接上图简单
- 适合监控点场景
- 但不如统一 geometry 字段规范

## 9. 当前结构中的注意点

### 9.1 危大作业存在新旧两套设计痕迹

当前危大作业相关表存在：

- `biz_danger_point`
- `biz_dangerous_work`

实际运行主线明显偏向 `biz_dangerous_work`，因此后续设计应以它为准。

### 9.2 项目分类字段存在“双表达”

项目表里同时存在：

- `project_type`
- `sub_type`

当前实际语义并不完全统一：

- `sub_type` 更像字典编码
- `project_type` 更像展示名称或业务类型文本

后续建议明确二者边界，避免查询和统计口径混乱。

### 9.3 附件表缺少物理外键

`biz_project_file` 当前只有索引，没有显式外键约束。业务可以运行，但数据库层一致性较弱。

### 9.4 地图标绘采用逻辑外键

`map_drawing.business_id` 是多态关联字段，灵活但不具备数据库强约束。后续若标绘量大或规则复杂，需要更明确的约束设计。

### 9.5 空间存储方式不完全统一

系统同时使用：

- PostGIS geometry
- GeoJSON 文本
- 经纬度字段

这对开发是方便的，但对长期维护来说会增加认知成本。

## 10. 建议的后续数据库优化方向

建议后续优先考虑以下方向：

1. 明确 `biz_danger_point` 与 `biz_dangerous_work` 的保留策略
2. 明确 `project_type` 与 `sub_type` 的职责边界
3. 为 `biz_project_file.project_id` 增加物理外键
4. 评估是否统一危大作业监控点为空间字段
5. 评估 `map_drawing` 是否需要更强的业务约束
6. 补充数据字典、状态码和值域说明
7. 为关键业务编码字段增加唯一索引或业务索引

## 11. 当前基线结论

当前数据库已经具备支撑系统继续迭代的核心条件：

- 用户权限模型已落表
- 项目主线业务已落表
- 字典与模板配置模型已落表
- 空间业务数据已落表
- 危大作业模块已具备独立数据结构
- 地图标绘与附件能力已具备数据支撑

但数据库当前也带有明显的“迭代过程痕迹”：

- 存在旧设计残留
- 存在部分逻辑关联未固化成外键
- 存在字段语义尚未彻底统一的情况

因此后续所有数据库迭代，应默认基于本文档继续收口和规范化。
