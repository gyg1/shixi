# 模块迭代路线与待办清单

## 1. 文档说明

本文档用于把当前系统现状转化为后续可执行的迭代路线和待办清单。

目标：

- 明确当前每个模块所处阶段
- 给出建议的迭代优先级
- 拆分出可执行的开发待办项
- 作为后续版本推进的任务基线

本文档以“先收口基础工程质量，再补齐核心业务，再做体验和治理增强”为原则进行排序。

## 2. 当前阶段判断

当前系统整体处于：

- 核心架构已完成
- 主要业务主线已跑通
- 地图能力较强
- 项目管理最成熟
- 配置驱动机制已建立
- 工程质量和模块边界仍需收口

更准确地说，系统当前不是“从 0 到 1”阶段，而是“从可运行到可维护、可扩展”的阶段。

## 3. 总体迭代策略

建议后续按四个阶段推进：

### 阶段一：工程基线收口

目标：

- 让前端类型检查通过
- 明确当前代码可构建基线
- 降低大文件耦合风险

### 阶段二：核心业务闭环增强

目标：

- 补齐项目管理细节
- 明确土地、房产、危大作业的后台管理边界
- 完善页面与接口联动

### 阶段三：系统治理与配置能力增强

目标：

- 统一请求层
- 完善系统设置
- 规范配置管理和权限边界

### 阶段四：体验、测试与部署规范化

目标：

- 建立测试基线
- 优化交互体验
- 形成可部署、可交接的工程状态

## 4. 模块优先级排序

建议优先级如下：

1. 前端工程基线
2. 地图主页面解耦
3. 项目管理模块
4. 危大作业模块
5. 土地储备模块
6. 房产资产模块
7. 字典与模板模块
8. 用户与权限模块
9. 系统设置模块
10. 测试与部署模块

- 形成稳定可检查、可继续开发的前端基线

- 修复 `UserMapView.vue` 中的 TypeScript 报错

- 修复 `ProjectList.vue` 中的可空值类型问题

## 8. 建议的最近三步

如果从现在开始实际推进，我建议最近三步直接按下面做：

1. 先修前端 `type-check`
2. 再拆 `UserMapView.vue`
3. 然后补齐项目管理细节

原因：

- 这是当前投入产出比最高的路径
- 能最快建立稳定开发面
- 后续再补土地、房产、危大作业会更顺

## 9. 当前基线结论

当前系统已经具备继续快速迭代的基础，但最关键的问题不是“缺页面”，而是：

- 工程基线还不稳
- 地图页耦合过重
- 若继续无序叠加功能，后续维护成本会迅速升高

因此建议后续迭代遵循以下原则：

- 先稳工程
- 再补主线
- 再扩模块
- 最后做治理和交付

后续所有开发任务，建议均基于本路线文档继续细化和更新。
