# 系统现状文档

## 1. 项目概述

本项目是一个基于 GIS 的三维城市规划协同管理系统，目标是将城市规划相关的业务数据、空间数据和统计分析能力统一到一个前后端分离的平台中，实现项目管理、土地储备、房产资产、危大作业监管、地图标绘和系统配置的集中管理。

当前项目已经完成基础架构搭建，并实现了较完整的前后端主流程，具备继续迭代和收敛的基础。

## 2. 项目目录结构

根目录主要包含以下内容：

- `CeuimDataScreen`
  - 前端项目，基于 Vue 3 + TypeScript + Vite
- `CeuimDataScreenSpringboot`
  - 后端项目，基于 Spring Boot + MyBatis-Plus + PostgreSQL/PostGIS
- `uploads`
  - 上传文件落盘目录，当前主要用于项目附件
- `start.md`
  - 课题背景、设计目标、阶段计划说明
- `agent.md`
  - 辅助说明文档

## 4. 系统角色与权限模型

系统当前包含三类角色：

- `USER`
  - 普通用户
  - 可登录并进入用户地图端
- `ADMIN`
  - 管理员
  - 可访问后台管理界面
  - 可管理项目、危大作业、字典、模板等业务数据
- `SUPER_ADMIN`
  - 超级管理员
  - 具备管理员全部权限
  - 额外拥有用户管理、系统设置等更高权限

权限控制是前后端双层实现：

- 前端通过路由守卫做页面访问控制
- 后端通过 Spring Security + JWT + `@PreAuthorize` 做接口控制

## 12. 当前已识别的问题与风险

### 12.1 前端大文件耦合严重

`UserMapView.vue` 体量非常大，承载了地图展示、业务面板、分析工具、危大作业展示、标绘等大量功能。后续继续迭代时，这个文件会成为高风险区域。

### 12.2 TypeScript 类型错误仍未清理

前端当前 `type-check` 不能通过，主要问题集中在：

- Cesium Viewer 构造参数类型不匹配
- `undefined / null` 可空值处理不完整
- 字典索引类型不明确
- 大量地图计算过程存在类型收口不足

这说明代码已经具备功能实现，但工程质量还需要继续收口。

### 12.3 前端请求层未统一

认证使用 `axios`，业务接口使用 `fetch`，后续会带来：

- 请求拦截逻辑分散
- 错误处理不一致
- Token 注入入口重复
- 维护成本增加

### 12.4 中文编码显示存在不稳定现象

从当前终端输出看，部分中文注释和字符串存在乱码现象，可能与文件编码、终端编码或历史拷贝过程有关。后续修改中文文案时需要谨慎处理。

### 12.5 配置硬编码较多

当前以下内容存在硬编码问题：

- API 地址
- 数据库账号密码
- Cesium Token
- 天地图 Token

后续部署到不同环境时需要做环境变量化。

### 12.6 测试体系不完整

前端虽然带有 Vitest 配置，但当前测试覆盖明显不足。后端也存在基础测试入口，但尚未形成稳定、可重复的完整测试体系。

项目当前所处阶段更准确地说是：

- 已有较多可运行功能
- 正处于功能收口、结构优化、工程化增强和模块补全阶段

## 14. 后续建议的迭代优先级

建议后续优先按以下顺序推进：

1. 修复前端 TypeScript 报错，建立稳定可构建基线
2. 拆分 `UserMapView.vue` 和地图相关大逻辑，降低耦合
3. 统一前端 API 请求层
4. 明确土地储备、房产资产、资金信息模块的后台页边界
5. 补齐接口文档、页面文档和数据库关系说明
6. 增加关键业务流程测试
7. 推进配置环境变量化和部署规范化

## 15. 当前基线结论

当前系统已经具备继续迭代开发的基础条件。后续所有开发建议默认基于以下事实展开：

- 项目是前后端分离架构
- 地图是系统核心交互载体
- 项目管理是当前最成熟的业务主线
- 系统已引入字典和表单模板作为业务扩展机制
- 后端已具备空间数据管理能力
- 当前最主要的工程问题在前端类型收口和大文件解耦

后续新增功能、重构或修复工作，建议均以本文件为基线同步更新。
